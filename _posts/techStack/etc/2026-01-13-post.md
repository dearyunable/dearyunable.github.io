Windows + WSL에서 Jekyll(minimal-mistakes) 블로그 만들기
(Ruby 설치부터 GitHub Pages 배포, 인증 오류 해결까지)

Windows 환경에서 Jekyll 블로그를 만들면서
WSL + Ruby + Bundler + minimal-mistakes + GitHub Pages까지
실제로 겪은 과정을 전부 정리한 기록이다.

1. 왜 Windows에서 바로 안 하고 WSL을 썼을까?

Jekyll은 Ruby 기반 정적 사이트 생성기다.
문제는 Ruby + Jekyll 조합이 Windows에서 굉장히 까다롭다는 것.

gem 설치 오류

native extension 빌드 실패

openssl / ffi / eventmachine 문제

환경 맞추다가 지침…

그래서 선택한 방식은 이거였다.

Windows는 도구만 사용하고
실제 개발 환경은 WSL(Ubuntu)로 통일

2. WSL 상태 확인 – Docker만 있고 Ubuntu는 없었다

PowerShell에서 확인:
```powershell
wsl --status
```

결과:

기본 배포: docker-desktop
기본 버전: 2


 WSL2는 설치돼 있었지만
 Ubuntu 배포판은 없는 상태

그래서 Ubuntu를 추가 설치했다.

3. Ubuntu 설치
```powershell
wsl --install -d Ubuntu
```

설치 후 최초 실행 시:

UNIX username

UNIX password 설정

이 계정은 WSL 전용 계정이며 Windows 계정과는 무관하다.

설치 확인:
```powershell
wsl -l -v
```
4. Ruby는 왜 rbenv로 설치했나

Ruby는 프로젝트마다 버전 요구사항이 다르다.
그래서 버전 관리 도구 없이 설치하면 바로 충돌이 난다.

선택: rbenv

rbenv의 역할

Ruby 여러 버전 설치 가능

프로젝트별 Ruby 버전 지정

시스템 Ruby와 완전 분리

5. WSL(Ubuntu)에 Ruby 설치
필수 패키지
```bash
sudo apt update
sudo apt upgrade -y

sudo apt install -y git curl build-essential \
libssl-dev libreadline-dev zlib1g-dev \
libyaml-dev libffi-dev libgdbm-dev libncurses5-dev
```
rbenv 설치
```bash
git clone https://github.com/rbenv/rbenv.git ~/.rbenv
echo 'export PATH="$HOME/.rbenv/bin:$PATH"' >> ~/.bashrc
echo 'eval "$(rbenv init -)"' >> ~/.bashrc
source ~/.bashrc
```
ruby-build 설치
```bash
git clone https://github.com/rbenv/ruby-build.git ~/.rbenv/plugins/ruby-build
```

Ruby 설치
```bash
rbenv install 3.3.4
rbenv global 3.3.4
ruby -v
```

6. Bundler와 Jekyll 이해하기
Bundler란?

Ruby 프로젝트의 의존성 관리자

Gemfile 기준으로 gem 버전을 고정 관리

Jekyll이란?

Markdown → HTML로 변환하는 정적 사이트 생성기

GitHub Pages와 궁합이 매우 좋음

Ruby가 설치된 곳(WSL)에 Bundler와 Jekyll도 같이 설치해야 한다.

7. Bundler & Jekyll 설치 (WSL에서)
```bash
gem install bundler
gem install jekyll
```

확인:
```bash
bundle -v
jekyll -v
```

8. Jekyll 기본 블로그 실행 (경고는 무시해도 된다)
```bash
jekyll new HelloBlog
cd HelloBlog
bundle install
bundle exec jekyll serve
```

Sass 관련 경고가 잔뜩 나왔지만:

에러 아님
실패 아님
deprecated warning (무시 가능)

브라우저 접속:

http://localhost:4000

9. minimal-mistakes 테마 적용

기본 테마(minima) 대신
실사용에 가장 많이 쓰이는 minimal-mistakes를 사용하기로 했다.

클론 위치 주의

기존 Jekyll 프로젝트 X
홈 디렉토리에서 별도 프로젝트로 클론 O
```bash
cd ~
git clone https://github.com/mmistakes/minimal-mistakes.git
cd minimal-mistakes
bundle
bundle exec jekyll serve
```

여기까지 해서 로컬 실행 성공

10. GitHub Pages 레포에 push하기 (여기서 진짜 삽질 시작)
에러 1: 비밀번호 인증 실패
Password authentication is not supported


GitHub는 이제 비밀번호 로그인 불가
Personal Access Token(PAT) 필요

에러 2: 403 Permission denied
Permission denied to dearyunable


토큰 권한 문제
cached credential 문제

해결:
```bash
git config --global --unset credential.helper
```
에러 3: workflow 파일 때문에 push 거부
refusing to allow a Personal Access Token to create or update workflow


원인:

.github/workflows/bad-pr.yml 존재

토큰에 workflow 권한 없음

해결:
```bash
rm -f .github/workflows/bad-pr.yml
git add -A
git commit -m "Remove bad workflow"
```
11. 최종 push 성공
```bash
git push -u origin master
```

결과:

* [new branch] master -> master
branch 'master' set up to track 'origin/master'


GitHub 업로드 완전 성공

12. GitHub Pages 설정

GitHub 레포 → Settings → Pages

Source: Deploy from a branch

Branch: master

Folder: / (root)

배포 주소:

https://dearyunable.github.io
